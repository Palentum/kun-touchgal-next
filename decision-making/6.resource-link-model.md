# 资源链接拆表策略

## 背景

当前 `patch_resource` 同时承担了两类职责：

1. 资源本身的信息
2. 资源下载链接的信息

这导致 `content` 可以塞入多条链接，但 `storage`、`size`、`code`、`password`、`hash` 只能保存一份，数据语义不一致。

网站现在计划将“资源”和“资源链接”正式拆开建模。

## 核心目标

1. 一个 `patch_resource` 可以包含多个独立下载链接
2. 每条下载链接都有自己独立的提取码、解压码、哈希、大小、存储类型
3. `patch_resource` 只保留资源级别信息
4. `patch_resource_link` 负责承载所有链接级别信息
5. 当前存量数据通过一次性迁移脚本完成迁移

## 决策

### 1. 移除 `patch_resource` 中现有的链接字段

**决策：直接从 `patch_resource` 中移除以下字段：**

- `storage`
- `size`
- `code`
- `password`
- `hash`
- `content`

原因：

- 这些字段本质上属于“单条下载链接”，不属于“资源主体”
- 当前结构无法正确表达“一个资源下多条链接、且每条链接参数不同”的场景
- 继续保留这些字段只会让模型长期处于半拆分状态

结论：

- `patch_resource` 只保留资源元信息
- 链接相关字段全部下沉到 `patch_resource_link`

### 2. 新增 `patch_resource_link` 模型

**决策：新增 `patch_resource_link`，并与 `patch_resource` 建立一对多关系。**

关系定义：

- 一个 `patch_resource` 是一个资源
- 一个 `patch_resource` 可以有多个 `patch_resource_link`
- 一个 `patch_resource_link` 只能属于一个 `patch_resource`

推荐字段：

- `id`
- `resource_id`
- `storage`
- `size`
- `code`
- `password`
- `hash`
- `content`
- `sort_order`
- `download`
- `created`
- `updated`

结论：

- `patch_resource` 负责资源卡片
- `patch_resource_link` 负责资源卡片下的多个下载入口

### 3. `patch_resource_link` 不需要 `status`

**决策：`patch_resource_link` 不新增 `status` 字段。**

原因：

- 当前资源审核、公开、删除等流程仍然以资源整体为单位
- 本次目标只是把“多个下载链接”建模正确，而不是引入“单链接审核/下架”能力
- 增加 `status` 会把本次改动范围扩展到更复杂的运营流程

结论：

- 资源是否可见仍由 `patch_resource.status` 控制
- 链接随所属资源一起生效

### 4. `patch_resource_link` 需要 `sort_order`

**决策：`patch_resource_link` 增加 `sort_order` 字段。**

原因：

- 一个资源下可能存在多个镜像或多个下载方式
- 前端展示需要稳定顺序，不能依赖数据库默认返回顺序
- 后续如果支持拖拽排序，`sort_order` 可以直接复用

结论：

- 前端与后端都应按 `sort_order asc` 读取资源链接

### 5. 存量数据通过一次性迁移脚本处理

**决策：不做双轨兼容，不保留旧字段过渡；直接通过迁移脚本完成旧数据搬迁。**

原因：

- 当前网站用户量较小，历史资源量有限
- 一次性迁移的复杂度低于长期维护双模型兼容
- 旧字段与新模型语义冲突明显，保留过渡层会拖慢后续改造

结论：

- 先改 schema
- 再执行一次迁移脚本
- 应用代码直接切换到新结构

### 6. 迁移脚本只负责数据搬运，不负责语义修复

**决策：迁移脚本只做结构迁移，不尝试“智能纠正”旧数据问题。**

迁移规则：

1. 遍历每条旧 `patch_resource`
2. 读取旧字段 `content`
3. 按既有规则拆分出多条链接
4. 为每条链接生成一个 `patch_resource_link`
5. 将原 `storage`、`size`、`code`、`password`、`hash` 写入对应 link
6. `sort_order` 按原始顺序递增写入

补充说明：

- 如果旧数据中的一条 `patch_resource` 存在多条链接，但只保存了一份提取码/解压码/哈希，这些值会被原样复制到拆分后的每个 link
- 如果旧 `content` 中包含多条链接，则这些链接的 `storage`、`size`、`code`、`password`、`hash` 全部继承原 `patch_resource` 上的同名属性
- 这是对历史数据结构限制的保守继承，不属于迁移错误

## 数据设计原则

### `patch_resource` 保留的职责

- 资源标题
- 资源备注
- 资源分类 `section`
- 资源标签维度 `type/language/platform`
- 所属用户
- 所属游戏
- 审核状态
- 点赞与聚合下载统计
- 创建与更新时间

### `patch_resource_link` 承担的职责

- 具体下载地址
- 存储类型
- 单链接文件大小
- 单链接提取码
- 单链接解压码
- 单链接哈希
- 单链接排序
- 单链接下载计数

## 代码迁移范围

本次改造需要同步调整以下区域：

- Prisma schema
- 资源创建接口
- 资源编辑接口
- 资源获取接口
- 资源发布表单
- 资源编辑表单
- 资源展示组件
- 资源下载计数逻辑
- 资源类型定义

## 非目标

本次决策不包含以下内容：

- 不做单链接审核
- 不做单链接单独下架
- 不做历史数据自动清洗
- 不做复杂双写兼容层

## 实施顺序建议

1. 新增 `patch_resource_link` 并调整 `patch_resource` schema
2. 更新 Prisma 类型与接口返回结构
3. 编写并执行一次性迁移脚本
4. 重构资源创建与编辑逻辑，改为读写 `links[]`
5. 重构资源展示逻辑，改为按 `links[]` 渲染
6. 清理所有旧字段使用点

## 总结

本次资源系统调整的最终结构为：

- `patch_resource` 是资源主体
- `patch_resource_link` 是资源主体下的多个下载链接
- 链接级字段全部下沉到 `patch_resource_link`
- `patch_resource_link` 不需要 `status`
- `patch_resource_link` 需要 `sort_order`
- 历史数据通过一次性迁移脚本完成搬迁
