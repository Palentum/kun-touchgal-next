# Steam 数据同步策略

## 背景

新增从 Steam 获取游戏标签、开发商和别名的功能，进一步丰富 patch 的元数据。

## 数据模型变更

在 `patch` model 新增字段：

| 字段       | 类型         | 说明         |
| ---------- | ------------ | ------------ |
| `steam_id` | Int?, unique | Steam App ID |

`patch_tag` 的 `source` 字段新增 `steam` 值。

## Steam ID 获取

### 从简介中提取

批量同步脚本 `migration/syncSteamId.ts` 从 patch 的 `introduction` 字段中提取 Steam 链接，解析出 App ID。

### 手动录入

创建和更新 Galgame 时，用户可手动填写 Steam ID。

## 同步内容

`ensurePatchDataFromSteam` 函数一次性处理三类数据：

### 1. 标签 (Tags)

- 创建 `patch_tag` 记录，`source: 'steam'`
- 使用 `lib/steamDirtyTag.ts` 中的 `lowQualitySteamTags` 列表过滤低质量标签
- 通过标签 `name` 去重，已存在同名标签仅建立关联

### 2. 开发商 (Companies)

- 从 Steam 数据中提取开发者和发行商
- 创建 `patch_company` 记录
- 建立 `patch_company_relation` 关联

### 3. 别名 (Aliases)

- 提取游戏的多语言名称
- 创建 `patch_alias` 记录，与已有别名去重

## 跳过条件

如果 patch 已有 `source: 'steam'` 的标签关联，直接跳过，不重复获取。

## 触发时机

- 创建 Galgame 时，若提供了 Steam ID
- 更新 Galgame 时，若提供了 Steam ID
- 在 VNDB tag 和 Bangumi tag 获取之后执行
- 批量同步脚本：`migration/syncSteamData.ts`

## 批量同步

`migration/syncSteamData.ts`：

- 并发数：2（避免 Steam API 限速）
- 处理所有有 `steam_id` 但尚无 Steam tag 的 patch
- 包含详细的进度日志

## 执行顺序

创建/更新 patch 时，各来源数据获取的执行顺序：

1. 用户手动添加的 tag（`handleBatchPatchTags`）
2. VNDB tag（`ensurePatchTagsFromVNDB`）
3. Bangumi tag（`ensurePatchTagsFromBangumi`）
4. Steam 数据（`ensurePatchDataFromSteam`，包含 tag、company、alias）
5. VNDB company（`ensurePatchCompaniesFromVNDB`）
6. DLsite company（`ensurePatchCompanyFromDlsite`）

## 相关文件

- `prisma/schema/patch.prisma` — 数据模型（`steam_id` 字段）

- `app/api/edit/create.ts` — 创建 patch 时调用
- `app/api/edit/update.ts` — 更新 patch 时调用
