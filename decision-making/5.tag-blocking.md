# 游戏标签屏蔽策略

## 背景

网站计划新增“游戏标签屏蔽”功能。

目标是：当用户屏蔽某个标签后，站内绝大多数公开列表页面都不再展示带有该标签的游戏，从而减少用户看到不感兴趣内容的概率。

同时，用户如果已经知道某个游戏的直达链接，仍然应当可以正常访问该游戏详情页。

## 核心目标

1. 用户可以主动屏蔽一个或多个标签
2. 被屏蔽标签对应的游戏，不再出现在站内公开列表中
3. 游戏详情页仍可正常访问
4. 屏蔽设置需要持久化
5. 过滤逻辑应尽量在服务端查询阶段完成，而不是前端拿到数据后再隐藏

## 决策

### 1. 仅登录用户可以屏蔽标签

**决策：只有已登录用户才允许使用标签屏蔽功能。**

原因：

- 标签屏蔽是一个明确的个人偏好设置，不属于匿名访客的临时浏览状态
- 该功能要求持久化，天然更适合作为用户设置的一部分
- 可以避免未登录用户依赖纯本地状态导致的行为不一致

### 2. 屏蔽设置需要持久化，Cookie 作为缓存

**决策：标签屏蔽数据需要持久化保存到用户维度；同时可以在 Cookie 中保留一份缓存，用于服务端快速读取。**

原因：

- 用户在不同会话下应保持相同的屏蔽结果
- 当前网站已有基于 Cookie 的内容过滤能力（如 NSFW 显示控制），标签屏蔽可以沿用相同思路
- 服务端列表页读取 Cookie 后直接拼接查询条件，能减少额外的用户设置查询开销

结论：

- 数据“真源”应为用户持久化设置
- Cookie 用作读取优化和服务端渲染缓存

### 3. 屏蔽标签后，所有公开列表页面都不显示对应游戏

**决策：只要某个游戏命中任意一个被屏蔽标签，该游戏就不应出现在站内公开列表结果中。**

适用范围包括但不限于：

- 首页推荐/最新内容
- 游戏列表页
- 搜索结果页
- 排行榜
- 标签页下的游戏列表
- 公司页下的游戏列表
- 随机游戏入口
- 与游戏绑定展示的资源列表

过滤原则：

- 命中任意一个屏蔽标签即可整体过滤该游戏
- 相关资源如果属于该游戏，也应一并隐藏

### 4. 游戏详情页仍可访问

**决策：标签屏蔽不会阻止用户访问游戏详情页。**

原因：

- 标签屏蔽的目标是“减少列表曝光”，不是“彻底封锁访问”
- 用户已经知道游戏链接或从站外进入时，仍应允许正常访问
- 该功能不应改变站点的资源可用性边界

结论：

- 详情页不做拦截
- 详情页 metadata 也不应被标签屏蔽逻辑阻断

### 5. 标签搜索可以搜到被屏蔽标签，但点进去后游戏列表为空

**决策：标签搜索结果仍然展示被屏蔽标签本身；但进入该标签详情页后，游戏列表会被过滤为空。**

原因：

- 用户需要有能力管理自己的屏蔽标签，搜索不到会影响取消屏蔽或确认状态
- 标签实体本身不属于需要被隐藏的数据，真正需要隐藏的是“带该标签的游戏”

结论：

- 搜索标签接口不因屏蔽而删除标签结果
- 标签详情页中的游戏列表查询仍要应用屏蔽过滤

### 6. 普通标签展示位不显示被屏蔽标签

**决策：在常规页面的标签展示区域，被用户屏蔽的标签本身不再显示。**

适用理解：

- 游戏卡片上的标签摘要不显示被屏蔽标签
- 其他普通标签展示区域也应遵循同样规则

原因：

- 如果游戏本身已经因命中屏蔽标签被过滤，那么其标签摘要自然不会出现
- 即使某个页面展示的是标签集合而非游戏列表，也应尽量避免继续暴露已屏蔽标签

补充说明：

- 标签搜索页是明确例外，仍然允许搜索到该标签

## 数据设计建议

推荐采用“用户持久化 + Cookie 缓存”的双层方案。

### 用户持久化

可选方案：

1. 新增用户屏蔽标签关系表
2. 在用户表上增加数组字段保存屏蔽标签 ID

倾向：

- 更推荐独立关系表，扩展性更好，也更方便后续统计、审计和去重

### Cookie 缓存

Cookie 中保存当前用户屏蔽标签 ID 列表，用于：

- SSR/Route Handler 直接读取
- 与现有 NSFW 过滤逻辑保持一致
- 降低每次公开列表请求都查询用户设置的成本

## 查询层实现原则

标签屏蔽应当作为一个统一的“内容可见性过滤条件”参与 `patch` 查询。

推荐将其抽象为公共查询条件，而不是分散在每个页面单独判断。

建议统一接入的查询入口：

- Galgame 列表查询
- 首页游戏查询
- 搜索查询
- 排行榜查询
- 标签关联游戏查询
- 公司关联游戏查询
- 随机游戏查询
- 资源列表中依赖 `patch` 可见性的查询

## 后台与详情页边界

### 详情页

- 不受标签屏蔽影响
- 用户仍可直接访问

### 后台

**决策：后台管理页面默认不受标签屏蔽影响。**

原因：

- 管理员需要看到完整数据
- 避免因为个人屏蔽配置导致后台数据缺失，增加排查成本

## 非目标

本次功能不包含以下目标：

- 不做“禁止访问某类游戏详情页”
- 不做“标签搜索结果隐藏被屏蔽标签”
- 不做“按设备分别保存不同标签屏蔽策略”
- 不做“管理端跟随个人屏蔽策略”

## 实施顺序建议

1. 设计并落地用户标签屏蔽持久化结构
2. 在 Cookie 中缓存屏蔽标签列表
3. 抽象统一的 `patch` 可见性过滤条件
4. 接入所有公开列表查询入口
5. 增加用户设置页的标签屏蔽管理入口
6. 增加标签页或标签相关区域的快捷屏蔽/取消屏蔽能力

## 总结

本次“标签屏蔽”功能的本质，不是隐藏标签实体本身，而是隐藏“命中该标签的游戏在公开列表中的曝光”。

最终规则为：

- 仅登录用户可使用
- 设置需要持久化
- Cookie 可作为缓存
- 被屏蔽标签对应的游戏不再出现在公开列表中
- 游戏详情页仍可正常访问
- 标签搜索仍可搜到被屏蔽标签
- 进入该标签页后，游戏列表应为空
- 后台默认不受该过滤影响
